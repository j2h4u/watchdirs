# Phase 03.2: Scan-Time Folder Collapse - Context

**Gathered:** 2026-06-15
**Status:** Ready for planning

<domain>
## Phase Boundary

At **scan time**, fold "never-interesting" subtrees into a SINGLE aggregate row ŌĆö keep the
collapsed directory's recursive size, stop emitting its descendant rows ŌĆö to cut directory
cardinality (the dominant driver of DB size) and the consuming agent's token cost.

The consumer of this tool is an **AI agent** investigating host disk growth; collapse decisions
are optimized for that consumer (interviewed during discussion), not a human.

**In scope:** scan-time collapse rules (which subtrees, what thresholds), the collapsed-row
schema, the `top_child` breadcrumb, and the `[collapse]` config surface + shipped defaults.

**Out of scope (explicitly):** report-time display controls ŌĆö `--depth` output cap, zero-delta
row suppression, prefix de-duplication, `--human` formatting. Those are a reporting-layer pass
(Phase 4 / future), NOT this phase. Retention/tiering = Phase 4. Differential/temporal storage =
deferred future phase.
</domain>

<decisions>
## Implementation Decisions

### Collapse mechanism
- **D-01:** Collapse happens at **scan/collect time** and is **irreversible** ŌĆö descendant rows
  of a collapsed subtree are never written. The collapsed directory's row keeps its full recursive
  aggregate (bytes, file_count). This is forward-only and matches the no-backward-compat policy;
  past snapshots' collapsed internals are unrecoverable, which is accepted (it is the storage point;
  current detail is always recoverable by a live re-scan with a tuned config).
- **D-02:** Collapse fires at the **shallowest** qualifying directory along a path (collapse the
  whole subtree at its entry node, not each descendant).

### Collapse triggers (a directory D is collapsed when ANY of these, AND D is not in `never`)
- **D-03:** **Known-noise name** ŌĆö `D`'s basename Ōłł `collapse.names`. reason = `known_noise`.
  (Locked baseline: cache/module/venv-type folders are always collapsed.)
- **D-04:** **Fan-out** ŌĆö `D`'s direct child-directory count Ōēź `collapse.fan_out` (**default 500**,
  the consuming agent's most-trusted signal). reason = `fan_out`.
- **D-05:** **Descendant-count safety net** ŌĆö `D`'s total descendant-dir count Ōēź
  `collapse.descendants` (**default 10000**) ŌĆö catches narrow-but-deep explosions that fan-out
  misses. reason = `descendant_count`.
- **D-06:** **Depth is NOT a collapse trigger.** The consuming agent ranked depth least-trustworthy
  ("deep paths are normal"). Depth belongs to report-time display, not storage.
- **D-07:** **`never_collapse` allowlist** ŌĆö `collapse.never` is a list of paths that are NEVER
  collapsed regardless of the above (protects real data dirs: `/var/lib/postgresql`, Maildirs,
  sharded object stores). Empty by default; operator curates.

### top_child breadcrumb
- **D-08:** When collapsing `D`, also record the **single largest direct child by size** as
  `top_child` (a `paths` reference + that child's bytes). One extra reference, not thousands ŌĆö
  preserves the agent's "is there something to drill into" signal; that child's growth is then
  visible across snapshots for free. **One** child, not top-N (Kaizen ŌĆö minimal).

### Collapsed-row schema (columns on `directory_sizes`, NULL for the ~99% non-collapsed rows)
- **D-09:** Add: `collapsed` (0/1, load-bearing), `collapse_reason`
  (`known_noise`|`fan_out`|`descendant_count`), `collapsed_dirs` (count of folded descendant dirs),
  `top_child_id` (REFERENCES paths(id), nullable). Existing size/file_count remain the recursive
  aggregate. Reporting omits NULL fields in JSON output.

### Config surface (`[collapse]` table in the TOML config), shipped defaults
- **D-10:** `names = [node_modules, .venv, .git, site-packages, __pycache__, .cache, .mypy_cache,
  .pytest_cache, .tox, .npm, .gradle, .cargo, .rustup]` ┬Ę `fan_out = 500` ┬Ę `descendants = 10000`
  ┬Ę `never = []`. Per-root overrides deferred (YAGNI) unless trivial.

### Claude's Discretion
- Exact placement of the collapse columns (inline on `directory_sizes` vs a side table) ŌĆö panel
  leaned inline (few collapsed rows, near-zero NULL cost); planner/researcher may revisit if the
  hot-table width matters.
- Precise scanner implementation of "collapse at shallowest qualifying node" and how the recursive
  aggregate is still summed without emitting descendant rows.
- Whether `descendant_count` is computed in the same walk pass or needs a second pass.
</decisions>

<specifics>
## Specific Ideas

- Consumer-agent interview (Sonnet, context-free) drove the signal ranking: **fan-out most trusted,
  depth least**; wants `collapsed:true` as the load-bearing field and a `top_child` drill breadcrumb;
  "one summary row, always" for big opaque trees.
- Measured impact (real /home, 108k dirs): collapse ŌåÆ ~19k dirs (5.6x); dictionary 24.7ŌåÆ4.4 MB;
  per-snapshot 8.1ŌåÆ1.4 MB; tiered-steady projection ~60 MB. Budget target ~50ŌĆō100 MB.
</specifics>

<deferred>
## Deferred Ideas

- **Report-time display controls** ŌĆö `--depth N` output cap (agent wanted default 3), zero-delta row
  suppression, parent-prefix de-duplication, bytes-vs-`--human` split, omit-NULL-fields in JSON.
  These are a reporting-layer pass ŌåÆ Phase 4 / future reporting work, NOT 03.2.
- **Growth-concentration collapse** ("don't collapse if one child holds >50% of the *growth*") ŌĆö
  impossible at scan time (delta is a report-time property). The `top_child` breadcrumb (D-08) is the
  scan-time stand-in. Revisit only if `top_child` proves insufficient.
- **Per-root collapse overrides** ŌĆö different thresholds per scan root. YAGNI for now.
- **Differential / temporal (SCD-2) storage** ŌĆö the bigger dedup lever; separate future phase, gated
  on a real `/` scan + measured day-over-day value churn.
- **Retention tiering** (dailyŌåÆweeklyŌåÆmonthly downsampling) ŌĆö Phase 4.

None of these block 03.2; collapse + Phase 4 tiering already hits the ~50ŌĆō100 MB target.
</deferred>
