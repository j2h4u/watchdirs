## Summary

The existing code already has the right seam for scan-time collapse: `scan_root()` is an iterative post-order traversal that accumulates each directory's recursive totals in `_Frame`, emits a `DirectoryAggregate` only when the frame is popped, and merges the finished child aggregate upward with `_merge_child()`. That means this phase does **not** need a new aggregation algorithm; it needs a row-emission policy layered on top of the current aggregation flow. [VERIFIED: src/watchdirs/collect/scanner.py + tests/test_scanner_semantics.py]

The most important planning finding is a migration hazard, not a scanner hazard. `initialize_database()` currently bumps `SCHEMA_VERSION` by replaying `schema.sql`, but `schema.sql` only uses `CREATE TABLE IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS`. If `SCHEMA_VERSION` is raised from `3` to `4` without an explicit migration path, an existing v3 database will be marked upgraded while `directory_sizes` still lacks the new collapse columns. The planner must therefore write a real v3→v4 migration path and must not rely on the current bootstrap logic for existing DBs. [VERIFIED: src/watchdirs/db/migrations.py + src/watchdirs/db/schema.sql][CITED: https://www.sqlite.org/lang_altertable.html][CITED: https://www.sqlite.org/faq.html]

The recommended implementation shape is: keep collapse metadata inline on `directory_sizes`, add an explicit additive migration, parse `[collapse]` in `config.py`, extend `ScannerOptions`, and keep reporting changes metadata-only. `query_top_rows()`, `query_diff_rows()`, and `query_explain_path_rows()` should surface that a boundary row is collapsed, but Phase 03.2 should not add report-time row suppression, depth caps, or alternative frontier logic. [VERIFIED: src/watchdirs/config.py + src/watchdirs/models.py + src/watchdirs/reporting/queries.py + src/watchdirs/reporting/render.py + .planning/phases/03.2-scan-time-folder-collapse/03.2-CONTEXT.md][ASSUMED]

**Primary recommendation:** Implement scan-time collapse as an extension of the current post-order scanner, ship inline `directory_sizes` metadata with an explicit v3→v4 `ALTER TABLE` migration, and extend reporting only enough to expose `collapsed` boundary facts and breadcrumbs. [VERIFIED: src/watchdirs/collect/scanner.py + src/watchdirs/db/migrations.py + src/watchdirs/reporting/queries.py][CITED: https://www.sqlite.org/lang_altertable.html][ASSUMED]

### Pattern 1: Post-Order Boundary Collapse Without Rewriting Aggregate Math
**What:** Keep the current iterative post-order walk and layer collapse onto row emission rather than replacing `_merge_child()` or aggregate accumulation. [VERIFIED: src/watchdirs/collect/scanner.py]
**When to use:** For all three triggers; this preserves current byte semantics and keeps hardlink, mount-skip, and partial-error behavior in one place. [VERIFIED: src/watchdirs/collect/scanner.py + tests/test_scanner_semantics.py][ASSUMED]
**Example:**
```python
# Source: src/watchdirs/collect/scanner.py (existing post-order walk), adapted for 03.2
row_start = len(rows)
frame.collapse_reason = early_reason(frame)  # basename or fan-out; may be None

# existing traversal still computes recursive bytes/file_count/dir_count
...
row = _directory_row(frame)
late_reason = frame.collapse_reason or descendant_reason(frame.dir_count)
if late_reason is not None:
    del rows[row_start:]          # remove any subtree rows already emitted
    row = mark_collapsed(row, late_reason, collapsed_dirs=frame.dir_count)
rows.append(row)
```
[VERIFIED: src/watchdirs/collect/scanner.py][ASSUMED]

**What:** Teach `initialize_database()` to alter a v3 DB in place before setting `user_version = 4`; do not depend on `schema.sql` replay to mutate existing tables. [VERIFIED: src/watchdirs/db/migrations.py + src/watchdirs/db/schema.sql]
**When to use:** Always for existing databases; the current bootstrap-only path is insufficient once `directory_sizes` needs extra columns. [VERIFIED: src/watchdirs/db/migrations.py][CITED: https://www.sqlite.org/lang_altertable.html]

- **Bumping `SCHEMA_VERSION` without a real migration:** Existing v3 DBs will be mis-labeled as upgraded while still missing columns. [VERIFIED: src/watchdirs/db/migrations.py + src/watchdirs/db/schema.sql]
- **Implementing collapse as report-time hiding only:** That saves output tokens but preserves the same stored cardinality. [VERIFIED: .planning/ROADMAP.md]

## Common Pitfalls

### Pitfall 1: Silent "Migration" That Does Not Alter the Table
**What goes wrong:** `SCHEMA_VERSION` is increased, `initialize_database()` runs, and the DB is marked upgraded even though `directory_sizes` still lacks `collapsed` fields. [VERIFIED: src/watchdirs/db/migrations.py + src/watchdirs/db/schema.sql]
**Why it happens:** The current bootstrap script only creates missing tables/indexes; it does not mutate an existing table definition. [VERIFIED: src/watchdirs/db/schema.sql][CITED: https://www.sqlite.org/lang_altertable.html]
**How to avoid:** Add a versioned migration branch for `user_version == 3` before the schema bootstrap returns. [VERIFIED: src/watchdirs/db/migrations.py][ASSUMED]
**Warning signs:** A v3 DB opens cleanly, `PRAGMA user_version` says `4`, and `SELECT collapsed FROM directory_sizes` fails. [VERIFIED: src/watchdirs/db/migrations.py][ASSUMED]

### Pitfall 2: Late Descendant Safety-Net Without Subtree Row Truncation
**What goes wrong:** `descendant_count` decides to collapse after children already emitted, but their rows remain in `rows`, so the subtree is both collapsed and fully expanded. [VERIFIED: src/watchdirs/collect/scanner.py][ASSUMED]
**Why it happens:** The total descendant count is only known at pop time, after the post-order subtree has been visited. [VERIFIED: src/watchdirs/collect/scanner.py]
**How to avoid:** Record the subtree's start index in the `rows` buffer and truncate that contiguous suffix when a late trigger fires. [VERIFIED: src/watchdirs/collect/scanner.py][ASSUMED]
**Warning signs:** A collapsed boundary row and one of its descendants both persist in the same snapshot. [ASSUMED]

### Pitfall 3: `top_child_id` Does Not Fully Satisfy the Locked Breadcrumb Contract
**What goes wrong:** The planner ships only `top_child_id`, but the context asked for the child reference **and that child's bytes**. [VERIFIED: .planning/phases/03.2-scan-time-folder-collapse/03.2-CONTEXT.md]
**Why it happens:** The named schema delta omits any `top_child_*_bytes` column, and the repo persists directories only. [VERIFIED: .planning/phases/03.2-scan-time-folder-collapse/03.2-CONTEXT.md + .planning/REQUIREMENTS.md]
**How to avoid:** Resolve this explicitly in planning: either add a numeric size field or narrow the requirement to a path-only breadcrumb before execution starts. [ASSUMED]
**Warning signs:** API payloads show a `top_child` path but cannot answer why that child was chosen. [ASSUMED]

### Pitfall 4: Render Layer Leaks Null Collapse Fields
**What goes wrong:** JSON includes `"collapse_reason": null` and similar keys on most rows, even though the context wants these omitted for non-collapsed rows. [VERIFIED: src/watchdirs/reporting/render.py + .planning/phases/03.2-scan-time-folder-collapse/03.2-CONTEXT.md]
**Why it happens:** Current render helpers always include some nullable keys like `error` and `group`, rather than conditionally inserting them. [VERIFIED: src/watchdirs/reporting/render.py]
**How to avoid:** Build collapse metadata keys conditionally in `_top_row_payload()`, `_diff_row_payload()`, and explain-path payloads. [VERIFIED: src/watchdirs/reporting/render.py][ASSUMED]
**Warning signs:** Snapshot JSON size barely drops even when row cardinality falls sharply. [ASSUMED]

## Open Questions (RESOLVED)

1. **Does `top_child` need a size column in addition to `top_child_id`?**
   - What we know: D-08 says "path reference + that child's bytes", but D-09 only names `top_child_id`. [VERIFIED: .planning/phases/03.2-scan-time-folder-collapse/03.2-CONTEXT.md]
   - Resolution: Phase 03.2 plans add `top_child_disk_bytes` alongside `top_child_id`. This preserves D-08's path-plus-bytes contract instead of narrowing it to a path-only breadcrumb. [RESOLVED: .planning/phases/03.2-scan-time-folder-collapse/03.2-01-PLAN.md]

2. **Should `top_child` mean direct child directory only?**
   - What we know: the product stores directory aggregates only, and `top_child_id` points into the directory `paths` table. [VERIFIED: .planning/REQUIREMENTS.md + src/watchdirs/db/schema.sql]
   - Resolution: `top_child` means the largest direct child directory by recursive disk bytes. The current model does not persist file-level breadcrumbs, and file inventory remains out of v1 scope. [RESOLVED: .planning/phases/03.2-scan-time-folder-collapse/03.2-02-PLAN.md]

3. **Should existing v3 databases be migrated or intentionally rebuilt?**
   - What we know: additive columns can be added cheaply in SQLite, and the current initializer cannot do that implicitly. [VERIFIED: src/watchdirs/db/migrations.py][CITED: https://www.sqlite.org/lang_altertable.html]
   - Resolution: existing v3 databases are explicitly migrated to v4. The plans require a transaction-scoped v3 to v4 SQLite migration before `PRAGMA user_version` advances, because retained history is the tool's value. [RESOLVED: .planning/phases/03.2-scan-time-folder-collapse/03.2-01-PLAN.md]
