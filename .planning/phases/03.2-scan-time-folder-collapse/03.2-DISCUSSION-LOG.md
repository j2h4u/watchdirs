## Methodology note

The product owner declined the standard gray-area interview, on the grounds that the tool's
**real user is an AI agent, not a human**. Instead:
1. A context-free **Sonnet** subagent was interviewed AS the consuming agent ("you investigate disk
   growth from this tool's JSON; what helps vs. wastes your tokens?").
2. Its answers were pressure-tested through a **5-voice expert panel** (Architect, Kaizen, QA,
   Product/agent-advocate, Sys Admin).
3. The owner chose to **trust Kaizen** (minimal scope) on the remaining open numbers.

## Collapse trigger for unknown trees (children vs depth)

| Option | Description | Selected |
|--------|-------------|----------|
| Depth of nesting | Collapse subtrees below a depth cap | |
| Fan-out (direct children) | Collapse a dir with ≥ N direct child dirs | ✓ |
| Descendant count | Collapse when total descendants ≥ N | ✓ (secondary safety net) |

**User's choice:** Fan-out, with a descendant-count safety net. **Depth rejected as a collapse
signal entirely.**
**Notes:** The consuming agent ranked fan-out most-trustworthy and depth least ("deep paths are
normal structure"). Fan-out default **500** (agent's number; Kaizen: don't over-think it).
Descendant safety net default **10000**. A `never_collapse` allowlist (QA) protects real data dirs.

## top_child breadcrumb

| Option | Description | Selected |
|--------|-------------|----------|
| None | Collapsed row stores only the aggregate | |
| top_child = 1 | Keep the single largest child by size | ✓ |
| top_child = 3 | Keep the top 3 | |

**User's choice:** One (Kaizen — minimal).
**Notes:** Preserves the agent's "is there something to drill into" signal at ~1 row cost; growth of
that child stays visible across snapshots. The agent's smarter "growth-concentration" rule needs
delta (report-time) and can't run at scan time — `top_child` (by size) is the scan-time stand-in.

## Phase scope — `--depth` / zero-delta suppression

| Option | Description | Selected |
|--------|-------------|----------|
| Include in 03.2 | Add report display controls now | |
| Defer to reporting | Keep 03.2 = storage-only; display controls later | ✓ |

**User's choice:** Defer (Kaizen — don't bloat the phase).
**Notes:** `--depth` and zero-delta suppression are report-time display, not scan-time storage.
Confusing `--depth` for a storage cap would catastrophically discard evidence (Sys Admin). 03.2 is
purely scan-time collapse; display controls go to the reporting layer / Phase 4.
