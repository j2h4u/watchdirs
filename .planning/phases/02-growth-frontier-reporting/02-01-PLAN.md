## Storage Domain Contract

- `mount` grouping label means the selected snapshot mount point that matched the row by segment-safe longest-prefix matching.
- `storage-domain` durable key is the tuple/string composed from `major_minor`, `root`, `filesystem_type`, and `mount_source`; it deliberately excludes `mount_id` because mount ids are snapshot-local and can be reused.
- `storage-domain` display label includes `filesystem_type`, `mount_source`, `major_minor`, `root`, and the selected `mount_point` when available, so later reports can distinguish the mounted storage object from the mount location label.
- `MountInfo.options` and `MountInfo.super_options` are intentionally not persisted in `SnapshotMount`: Phase 2 grouping uses identity and label fields only, while options are volatile policy/configuration details that can change without changing the storage domain.

This plan creates or extends these SQL tables and indexes:

- `snapshot_mounts`
- `snapshot_mounts_snapshot_idx`
- `snapshot_mounts_snapshot_mount_point_idx`
- `snapshot_mounts_snapshot_domain_idx`

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-02-01 | Spoofing | `snapshot_mounts` grouping identity | mitigate | Persist `major_minor`, `root`, `mount_point`, `filesystem_type`, and `mount_source`; do not use `mount_id` as a durable cross-snapshot key. |
| T-02-02 | Tampering | BLOB mount paths | mitigate | Store `root` and `mount_point` as BLOB values through `sqlite3.Binary`, matching Phase 1 path identity. |

| T-02-04 | Denial of Service | mount row volume | accept | Mountinfo rows are bounded by the local mount table and stored once per snapshot; no recursive live scan is introduced here. |
