# Phase 03.1: Storage Efficiency - Research

**Researched:** 2026-06-14
**Domain:** SQLite schema normalization (path dictionary), storage benchmark methodology, `collect` observability
**Confidence:** HIGH (most claims verified empirically against the host runtime; the locked architecture is not relitigated)

## Summary

The architecture is locked (D-01..D-11): a flat full-path dictionary in SQLite, no engine
change, a Hybrid A+C benchmark, a per-snapshot byte-budget gate, and `logging`→stderr
observability. This pass is an implementation-detail audit that ran the actual host SQLite
and the actual codebase to find where the locked HOW needs adjustment.

Three findings change what the planner must write (none change the locked WHAT):

1. **The D-04 dedup SQL is wrong as written.** `INSERT OR IGNORE INTO paths(path) … RETURNING id`
   returns **nothing** when the path already exists (the conflict is suppressed, so no row is
   "directly modified", so RETURNING is empty). Verified empirically on this host and confirmed
   by official SQLite docs. The robust pattern is an in-process `dict[bytes,int]` cache plus a
   **SELECT-on-miss, INSERT-on-true-miss** flow (details in Pitfall 1). The WHAT (insert-time
   dedup via a cache) is unchanged; only the SQL changes.

2. **D-02 "store as TEXT" cannot mean "decode paths to `str`".** The entire pipeline is raw
   filesystem **bytes** (paths are not guaranteed UTF-8). Binding a surrogateescape `str` to a
   column raises `UnicodeEncodeError`; binding `bytes` to a `TEXT`-declared column stores them
   losslessly but with `typeof = blob`. So the planner should declare the column `TEXT` (for
   collation/`LIKE`/GLOB intent) but **keep binding `bytes`** exactly as today. The reporting
   layer keeps receiving `bytes` and its Python prefix/ancestor logic is untouched. This
   reconciles D-01/D-02 with reality.

3. **The dictionary's `paths.path UNIQUE` constraint creates an autoindex that stores the path
   bytes a second time.** Verified via `dbstat`: `sqlite_autoindex_paths_1` is as large as the
   table itself. The dictionary still wins big (paths stored ~2× total instead of ~412× across
   snapshots), but the benchmark must count the autoindex in the "after" size, and the gate must
   be set against that real number — not an idealized "paths stored once".

The host has **two** Python/SQLite stacks: system `python3` → sqlite 3.46.1, and the
`uv`-managed venv (what the tool and tests actually run under) → sqlite **3.50.4**. Both are far
above the 3.35.0 RETURNING floor and both compile `ENABLE_DBSTAT_VTAB`. Tests/tool run via `uv`
(see test-invocation memory), so 3.50.4 is the binding runtime.

**Primary recommendation:** Land the flat dictionary with a `dict[bytes,int]` cache +
SELECT-on-miss/INSERT pattern (NOT `OR IGNORE … RETURNING`), declare `paths.path` as `TEXT` but
keep binding `bytes`, drop the `name` column, int-ize the three path indexes, set
`page_size`/`auto_vacuum` on a virgin connection before the first table and `VACUUM` to apply.
Sequence the **churn measurement first** (it sets the gate number), and gate cold-cache timing
behind a `sudo`-or-skip path because `/proc/sys/vm/drop_caches` is root-only on this host.

### Pattern 1: Cache + SELECT-on-miss path-id resolution (replaces broken D-04 SQL)

**What:** Resolve a path to its `path_id` using an in-process dict, falling back to SELECT then
INSERT only on a true miss. This preserves D-04's intent (in-process cache, ~100% hit after
snapshot #1) while avoiding the RETURNING gotcha and avoiding a write per duplicate.

**When to use:** Inside `insert_directory_rows`, before binding `directory_sizes` rows.

```python
# Source: verified on host sqlite 3.50.4; pattern per SQLite RETURNING docs
# https://www.sqlite.org/lang_returning.html ("only returns rows directly modified")
def _resolve_path_id(connection, cache: dict[bytes, int], path: bytes) -> int:
    cached = cache.get(path)
    if cached is not None:
        return cached
    row = connection.execute(
        "SELECT id FROM paths WHERE path = ?", (path,)
    ).fetchone()
    if row is not None:
        path_id = int(row[0])
    else:
        cur = connection.execute(
            "INSERT INTO paths(path) VALUES (?)", (path,)
        )
        path_id = int(cur.lastrowid)
    cache[path] = path_id
    return path_id
```

Notes:
- Bind **`bytes`** (the raw path), not a decoded `str` — see Pitfall 2.
- The cache key is `bytes`; hit-rate approaches 100% after snapshot #1 (D-04), so the SELECT
  fires only for genuinely new paths — ingestion stays fast.
- `cur.lastrowid` is the correct id for a plain `INTEGER PRIMARY KEY` (rowid alias, D-02).
- Build the full `(path_id, parent_id)` set first, then `executemany` the `directory_sizes`
  batch — keep the existing `INSERT_BATCH_SIZE`/`executemany` structure (reusable asset).

### Pattern 2: int-equality diff JOIN (D-01)

**What:** The cross-snapshot diff joins on `path_id` integers and joins `paths` once to
materialize the byte path for the Python layer.

**When to use:** Rewrite of `query_diff_rows` in `reporting/queries.py`.

```sql
-- Source: EXPLAIN QUERY PLAN verified on host -> covering-index searches (faster than BLOB)
WITH all_ids AS (
    SELECT path_id FROM directory_sizes WHERE snapshot_id = :baseline_id
    UNION
    SELECT path_id FROM directory_sizes WHERE snapshot_id = :current_id
)
SELECT
    p.path AS path,
    COALESCE(cp.path, pp.path) AS parent_path,   -- via parent_id -> paths join
    COALESCE(curr.depth, prev.depth) AS depth,
    COALESCE(prev.disk_bytes, 0) AS previous_disk_bytes,
    COALESCE(curr.disk_bytes, 0) AS current_disk_bytes,
    ...
FROM all_ids a
JOIN paths p ON p.id = a.path_id
LEFT JOIN directory_sizes prev ON prev.snapshot_id = :baseline_id AND prev.path_id = a.path_id
LEFT JOIN directory_sizes curr ON curr.snapshot_id = :current_id  AND curr.path_id = a.path_id
LEFT JOIN paths pp ON pp.id = prev.parent_id
LEFT JOIN paths cp ON cp.id = curr.parent_id
ORDER BY (COALESCE(curr.disk_bytes,0) - COALESCE(prev.disk_bytes,0)) DESC, depth DESC, p.path ASC
```

Verified `EXPLAIN QUERY PLAN` output: `SEARCH prev USING COVERING INDEX ds_pathid_snap`,
`SEARCH curr USING COVERING INDEX ds_pathid_snap`, `SEARCH p USING INTEGER PRIMARY KEY`. The
diff is faster than today's BLOB-equality version (D-01 claim confirmed).

### Anti-Patterns to Avoid

- **`INSERT OR IGNORE … RETURNING id` for dedup** — returns nothing on the (dominant) duplicate
  case. Verified broken. (Pitfall 1.)
- **Decoding paths to `str` to satisfy "TEXT"** — raises `UnicodeEncodeError` on non-UTF-8
  paths. Keep binding `bytes`. (Pitfall 2.)
- **Setting `page_size`/`auto_vacuum` after data exists and skipping VACUUM** — the pragma is
  silently ignored / file only grows. (Pitfall 3.)
- **Comparing two DBs with different page_size/journal_mode/auto_vacuum** — invalidates the
  size comparison (D-10). Both benchmark DBs must share identical PRAGMAs.
- **Writing progress/ETA to stdout** — pollutes the JSON contract. stderr only (D-11).
- **Adding a migration/compat shim for the old schema** — forbidden by project rules; bump
  `SCHEMA_VERSION`, ship fresh schema, regenerate dev DBs.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Per-object DB size attribution | Manual page walking / row-size estimation | `dbstat` vtable | Built into host sqlite; exact page accounting incl. hidden autoindex |
| Total file size truth | Trust `os.path.getsize` alone | `page_count*page_size` AND `stat()` post-VACUUM (assert equal) | Cross-check catches WAL/uncheckpointed pages skewing the number |

**Key insight:** SQLite already knows its own byte layout precisely (dbstat) and its own
id-generation (lastrowid). Re-deriving either by hand reintroduces exactly the
estimate/eyeball error this phase exists to eliminate.

## Common Pitfalls

### Pitfall 1: `INSERT OR IGNORE … RETURNING id` returns nothing on duplicates (breaks D-04)

**What goes wrong:** The proposed dedup SQL silently returns an empty cursor for every path
that already exists in `paths` (i.e. nearly every row after snapshot #1). Code that expects a
row gets `None` and either crashes or inserts NULL `path_id`.
**Why it happens:** `OR IGNORE` suppresses the UNIQUE conflict, so no row is "directly
modified", and per the SQLite RETURNING contract RETURNING only reports directly-modified rows.
**How to avoid:** Use the dict-cache + SELECT-on-miss + plain-INSERT pattern (Pattern 1). If a
single-statement upsert is genuinely wanted, `ON CONFLICT(path) DO UPDATE SET path=excluded.path
RETURNING id` does return the id on both new and duplicate — but it performs a redundant write
on every duplicate, which contradicts D-04's ingestion-speed rationale.
**Warning signs:** `path_id` columns coming out NULL; `fetchone()` returning `None` for known
paths; ingestion slower (not faster) than the BLOB version.
**Evidence (this host, sqlite 3.50.4):**
```
INSERT OR IGNORE new:   returned=[(1,)]
INSERT OR IGNORE dup:   returned=[]        # the gotcha
DO NOTHING dup:         returned=[]        # same gotcha
DO UPDATE dup:          returned=[(2,)]    # works but writes every dup
```
Confirmed by docs: "The RETURNING clause only returns rows that are directly modified."
[CITED: sqlite.org/lang_returning.html]

### Pitfall 2: "TEXT" column + non-UTF-8 paths

**What goes wrong:** Declaring `paths.path TEXT` and then decoding the raw byte path to `str`
(via `os.fsdecode`, which uses surrogateescape) and binding the `str` raises
`UnicodeEncodeError: surrogates not allowed` when sqlite tries to encode it.
**Why it happens:** Filesystem paths are arbitrary bytes; not all are valid UTF-8. sqlite's
`str` binding requires valid UTF-8.
**How to avoid:** Keep binding **`bytes`**. A `bytes` value bound to a `TEXT`-declared column is
stored losslessly (roundtrip verified) but with `typeof = blob`. Declaring the column `TEXT`
still buys the *intent* (collation, `LIKE`/`GLOB` semantics for the ASCII-path common case) and
matches D-02's wording, while the storage stays byte-exact. The reporting layer keeps doing
`bytes(row["path"])` — unchanged.
**Warning signs:** `UnicodeEncodeError` during `collect` on hosts with non-UTF-8 filenames;
`typeof(path)` surprising a reader who assumed `text`.
**Evidence (this host):**
```
bytes  -> TEXT col:  typeof=blob,  roundtrip == original: True
fsdecode(str) -> TEXT col:  UnicodeEncodeError: surrogates not allowed
```

### Pitfall 3: page_size / auto_vacuum applied too late

**What goes wrong:** Setting `PRAGMA page_size` or `auto_vacuum` after the first table/write has
no effect; the file keeps the default 4096 page size / `auto_vacuum=0`, and the benchmark's
"after" DB is mis-tuned, invalidating the comparison.
**Why it happens:** Both pragmas are file-format properties fixed at first write. They only
change later via a `VACUUM`.
**How to avoid:** Set both on a **virgin connection before any table** (Pattern 4). For an
already-populated DB, set the pragma then `VACUUM` to materialize it.
**Warning signs:** `PRAGMA page_size` / `PRAGMA auto_vacuum` returning the default after you
set them; benchmark "after" file unexpectedly large.
**Evidence (this host):**
```
auto_vacuum set before WAL & first table:  1 (FULL, OK)
auto_vacuum set on populated db, no VACUUM: 0 (ignored)
auto_vacuum set on populated db, then VACUUM: 1 (applied)
```

### Pitfall 4: Invalid size comparison (mismatched PRAGMAs / uncheckpointed WAL)

**What goes wrong:** Comparing OLD vs NEW DBs that differ in page_size/journal_mode/auto_vacuum,
or reading size while WAL pages are uncheckpointed, yields a wrong delta.
**How to avoid:** Build both benchmark DBs with identical PRAGMAs (D-10). Before measuring,
`VACUUM` (or checkpoint + truncate WAL) and assert `page_count*page_size == os.path.getsize()`.
**Warning signs:** The two reconciliation numbers disagree; ratio changes between runs.

### Pitfall 5: `drop_caches` needs root on this host

**What goes wrong:** `echo 3 > /proc/sys/vm/drop_caches` from the tool user fails (permission
denied), so the "cold" timing is actually warm.
**Why it happens:** `/proc/sys/vm/drop_caches` is `--w------- root root` (mode 0200) here.
**How to avoid:** Run the cold step via `sudo sh -c 'echo 3 > /proc/sys/vm/drop_caches'`
(operator is in the `sudo` group) and **report "cold unavailable, warm only" if sudo is absent**
rather than silently mislabeling. Alternative: open the DB file and call
`os.posix_fadvise(fd, 0, 0, os.POSIX_FADV_DONTNEED)` to evict just that file's pages without
root — narrower (only the DB, not the FS metadata) but unprivileged.
**Warning signs:** Cold and warm medians nearly identical; permission errors in the harness.

### Pitfall 6: The UNIQUE autoindex doubles path storage in the "after" DB

**What goes wrong:** Estimating the dictionary win as "paths stored once" overstates the gain.
`paths.path UNIQUE` creates `sqlite_autoindex_paths_1` which stores the path bytes again.
**Why it happens:** A UNIQUE constraint is backed by an implicit index containing the indexed
column.
**How to avoid:** The benchmark already measures real size via dbstat, so this is captured
automatically — but the planner must set the D-09 budget against the **measured** "after" size
(table + autoindex + int FKs in directory_sizes + int indexes), not an idealized model. The win
is still large: path bytes go from ~412× (per-snapshot) to ~2× (table + autoindex) total.
**Evidence (this host):** `sqlite_autoindex_t_1` measured equal in bytes to the table itself.

### Churn measurement on the EXISTING schema (the ROI-determining first deliverable, D-08/D-09)

```python
# Source: verified on host; runs against the CURRENT blob schema, before the rewrite.
# For each consecutive (prev, curr) snapshot pair of the same root:
CHURN_SQL = """
SELECT
  (SELECT COUNT(*) FROM directory_sizes WHERE snapshot_id = :prev) AS rows_prev,
  (SELECT COUNT(*) FROM directory_sizes WHERE snapshot_id = :curr) AS rows_curr,
  (SELECT COUNT(*) FROM (
      SELECT path FROM directory_sizes WHERE snapshot_id = :curr
      EXCEPT
      SELECT path FROM directory_sizes WHERE snapshot_id = :prev)) AS new_paths,
  (SELECT COUNT(*) FROM (
      SELECT path FROM directory_sizes WHERE snapshot_id = :prev
      EXCEPT
      SELECT path FROM directory_sizes WHERE snapshot_id = :curr)) AS deleted_paths
"""
# Cardinality / dedup potential across ALL snapshots of a root:
CARDINALITY_SQL = """
SELECT
  (SELECT COUNT(DISTINCT path) FROM directory_sizes) AS distinct_paths,
  (SELECT COUNT(*)            FROM directory_sizes) AS total_rows
"""
# churn_rate = new_paths / rows_curr ;  dedup_ratio = total_rows / distinct_paths
# Verified example (5% synthetic churn, 1000 dirs):
#   rows_prev=1000 rows_curr=1000 new=50 deleted=50 distinct=1050 total=2000
#   churn=0.050   dedup_ratio=1.90x   (this ratio IS the ROI driver per the panel)
```
Run this on 3–5 spaced real scans (D-08 method C). High dedup_ratio / low churn ⇒ ~5–6× win;
high churn ⇒ win collapses toward index int-ization. Use the **measured** churn to drive method
A's synthetic replication and to set the D-09 byte budget.

### collect observability: stderr logging, ETA, structured summary (D-11)

```python
# Source: stdlib logging; cli.py already keeps stdout=JSON (emit_json) and uses sys.stderr for errors.
import logging, sys, time

logger = logging.getLogger("watchdirs.collect")

def configure_collect_logging(verbose: bool) -> None:
    handler = logging.StreamHandler(sys.stderr)   # NEVER sys.stdout (keeps JSON contract pure)
    handler.setFormatter(logging.Formatter("%(asctime)s %(levelname)s %(message)s"))
    logger.addHandler(handler)
    logger.setLevel(logging.INFO if verbose else logging.WARNING)

# ETA from dirs-scanned rate during traversal:
start = time.monotonic()
def log_progress(dirs_done: int, dirs_total_estimate: int | None) -> None:
    elapsed = time.monotonic() - start
    rate = dirs_done / elapsed if elapsed > 0 else 0.0
    eta = ((dirs_total_estimate - dirs_done) / rate) if (dirs_total_estimate and rate) else None
    logger.info("scanned %d dirs, %.0f dirs/s%s",
                dirs_done, rate,
                f", ETA {eta:.0f}s" if eta is not None else "")

# One structured end-summary record (field set is Claude's discretion per D-11):
def log_summary(dirs: int, duration_s: float, db_bytes: int) -> None:
    logger.info("collect summary dirs=%d duration_s=%.2f db_bytes=%d", dirs, duration_s, db_bytes)
```
Notes:
- `scan_result.row_count` (already computed in `scanner.scan_root`) is the natural `dirs` count
  for the summary — no extra bookkeeping needed.
- A total-dir estimate for ETA is not known a priori in a single-pass `os.scandir` walk; either
  report rate-only (no ETA) on the first scan, or seed `dirs_total_estimate` from the previous
  snapshot's `row_count` for the same root (cheap, monotone-ish, good enough for an ETA hint).
- `db_bytes` for the summary should be read AFTER commit via `page_count*page_size` (or
  `os.path.getsize` of the DB) — note this is the live WAL-mode size, not a VACUUMed size.
