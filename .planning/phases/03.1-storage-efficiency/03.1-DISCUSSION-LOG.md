## Dictionary shape

| Option | Description | Selected |
|--------|-------------|----------|
| Flat full-path dictionary | `paths(id, path UNIQUE)`; rows reference `path_id`. Simple joins, faster int-equality diff. ~5–6×. | ✓ |
| Prefix/tree (`parent_id`) | Dedups shared prefixes too; recursive CTE to rebuild full paths. ~6–7%. | |
| Flat now, tree later | Implement flat, escalate to tree only if it misses target. | (subsumed) |

**Panel's choice:** Flat.
**Notes:** Architect — all hot reporting logic consumes full byte paths in Python, so tree
forces recursive path reconstruction on every report (worse CPU + 5-file rewrite) for only
~15–20% extra dedup. Flat also makes the cross-snapshot diff JOIN faster (int vs BLOB
equality). `name` column dropped (no readers); `parent_id` kept as int for the diff COALESCE.

## Size-target gate

| Option | Description | Selected |
|--------|-------------|----------|
| Absolute cap (≤2 GB) | Hard ceiling on retention-scale DB. | |
| Reduction ratio (≥Nx) | Gate on before/after ratio. | |
| Per-snapshot byte budget | ≤ X MB/snapshot at measured churn. | ✓ |

**Panel's choice:** Per-snapshot byte budget (ratio quoted as color).
**Notes:** QA — absolute cap is brittle if retention count changes; ratio is gameable (inflates
under the unrealistic 0%-churn synthetic). Per-snapshot budget is linear in the retention knob
and is what method A directly measures. Concrete number set after the real scan reveals dir
count + churn.

## Benchmark methodology

| Option | Description | Selected |
|--------|-------------|----------|
| Synthesize retention-scale DB (A) | One real scan replicated into ~412 snapshots, both schemas. | (spine) |
| Single scan, extrapolate (B) | Arithmetic extrapolation. | |
| Real multi-scan over time (C) | Several real spaced scans for true churn. | (bolt-on) |
| **Hybrid A+C** | A's replication driven by churn measured from 3–5 real spaced scans. | ✓ |

**Panel's choice:** Hybrid A+C.
**Notes:** QA — A alone silently models 0% churn (412 identical snapshots), making the
dictionary look maximally good; that is the lie to guard against. Measure real churn (C), feed
it into A. Reject B (eyeballed-estimate trap). Measure size via VACUUM-first + dbstat
table/index split, identical PRAGMAs; scan time cold vs warm under prod niceness, median ≥3.

## collect observability

| Option | Description | Selected |
|--------|-------------|----------|
| Python logging → journal | INFO progress/ETA lines + structured end summary; stdout stays JSON. | ✓ |
| Structured JSON progress events | Machine-readable progress records throughout. | (end-summary only) |
| Both | Human INFO during + structured summary at end. | ✓ (merged) |

**Panel's choice:** logging → stderr/journal, human progress + ETA during, structured summary at end.
**Notes:** Sys Admin — under Phase 4's root systemd unit, stderr logging lands in the journal
for free; stdout must stay pure JSON. ETA from dirs-scanned rate.

## Engine choice (raised by panel, not originally a question)

**Panel's choice:** SQLite only; DuckDB/SurrealDB shelved.
**Notes:** Researcher — DuckDB's headline compression is measured vs *denormalized* data; most
of its edge is the dictionary encoding we're doing ourselves, while SQLite keeps a 25-year
stable single-file forensic format. SurrealDB's LSM/versioning adds disk. zstd VFS extensions
rejected (immature). Escalate only if the budget gate fails by a wide margin.
