# Phase 03.1: Storage Efficiency - Context

**Gathered:** 2026-06-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Shrink `watchdirs`'s **own** on-disk SQLite footprint so the tool does not bloat the
disk it exists to protect. Today `directory_sizes` stores `path` / `parent_path` /
`name` as inline BLOBs in every row, repeated per snapshot, and 3 indexes also carry
the path BLOB. At Phase 4 retention scale (~412 snapshots × ~50k dirs) this was roughly
(unmeasured) ~8 GB.

This phase delivers: (1) a path-dictionary schema, (2) a **real measured** before/after
size + scan-duration benchmark, and (3) `collect` observability (progress + ETA +
duration). It is a hard prerequisite for Phase 4 scheduling — no scheduled/prod
collection may run until the dictionary lands and the benchmark passes.

**In scope:** schema normalization (path dictionary), benchmark harness/methodology,
collect logging. **Out of scope:** scheduled collection, retention/TTL pruning, timer
units (all Phase 4); engine migration (only if the gate below fails).

</domain>

<decisions>
## Implementation Decisions

Decisions below were produced by an engineering panel (System Architect, Researcher,
QA/Benchmark, Sys Admin, Kaizen) convened during discussion. Three deep experts
independently converged on the same approach and the same dominant risk.

### Dictionary shape — FLAT, in SQLite
- **D-01:** Use a **flat full-path dictionary**: `paths(id INTEGER PRIMARY KEY, path TEXT UNIQUE)`;
  `directory_sizes` references `path_id`. NOT a prefix/tree (`parent_id`) shape.
  Rationale: every hot reporting path (mount-prefix matching, ancestor tests, subtree
  scans in `reporting/` + `diagnostics/`) consumes the **full byte path in Python**, not
  SQL. A tree shape forces recursive-CTE path reconstruction on every report —
  reintroducing the bytes we saved with worse CPU and a 5-file rewrite — for only ~15–20%
  extra dedup. Flat is near-drop-in: add one `JOIN paths` per SELECT; the cross-snapshot
  diff JOIN becomes int-equality (`prev.path_id = curr.path_id`), which is *faster* than
  BLOB equality.
- **D-02:** Store paths as **TEXT, not BLOB** (BLOB buys nothing here and loses
  collation/`LIKE`). `id` as plain `INTEGER PRIMARY KEY` (free rowid alias).
- **D-03:** **Drop the `name` column** entirely — zero readers in current code. **Keep
  `parent_id` as an int FK** into `paths` only to preserve the diff query's `COALESCE`
  convenience. The 3 path-bearing indexes become int-keyed (`(path_id, snapshot_id)`,
  `(snapshot_id, parent_id)`) — this index int-ization is a large share of the win.
- **D-04:** Insert-time dedup via an in-process `dict[bytes,int]` cache during scan +
  `INSERT OR IGNORE INTO paths(path) … RETURNING id`. Paths are stable across snapshots,
  so cache hit-rate approaches ~100% after snapshot #1 — this also speeds ingestion.
- **D-05:** Apply `PRAGMA page_size` (tuned) **before** the rebuild, `VACUUM` to apply
  (mandatory — the file otherwise only grows), set `PRAGMA application_id` as a forensic
  file-type magic number.

### Engine choice — SQLite only (no migration now)
- **D-06:** **No engine change.** DuckDB/SurrealDB stay shelved. The Researcher found
  DuckDB's headline compression is measured against *denormalized* data — most of its
  advantage IS the dictionary encoding we are doing ourselves in SQLite, while SQLite
  keeps a 25-year-stable single-file forensic format. SurrealDB's LSM/versioning *adds*
  disk. Immature zstd VFS extensions are rejected for a forensic tool.
- **D-07:** Escalation to DuckDB is considered **only if** the benchmark misses the size
  gate (D-09) by a wide margin. PROJECT.md currently lists DuckDB/SurrealDB as
  out-of-scope for v1 — revisit there only if escalation actually triggers.

### Benchmark methodology — Hybrid A+C
- **D-08:** **Hybrid A+C.** Spine (A): run ONE real full scan, replicate rows into ~412
  synthetic snapshots under BOTH old-BLOB and new-dictionary schemas, `VACUUM`, compare
  file sizes. Bolt-on (C): run **3–5 real scans spaced over time** to measure the actual
  path **churn rate** (fraction new/deleted between consecutive snapshots) and drive A's
  replication with measured churn instead of assuming stability. Reject (B) pure
  arithmetic extrapolation (the eyeballed-estimate trap the user already rejected).
- **D-09:** **PASS/FAIL gate = per-snapshot byte budget** at *measured* churn (e.g.
  ≤ X MB/snapshot). Chosen over an absolute cap (brittle if retention count changes) and a
  reduction ratio (gameable — inflates under the unrealistic 0%-churn synthetic). Quote
  the ratio as color, gate on the budget. The concrete budget number is set once the real
  scan reveals dir count + churn.
- **D-10:** Measure size correctly: `VACUUM` first; report `page_count*page_size` AND raw
  `stat()` bytes (must match post-VACUUM); break out **table-vs-index via `dbstat`**.
  Identical PRAGMAs (page_size, journal_mode, auto_vacuum) on both DBs or the comparison
  is invalid. Scan wall-clock: cold-cache (`drop_caches`) vs warm reported separately,
  under prod `nice`/`ionice -c2 -n7`, median of ≥3 runs + spread, root-by-root.

### collect observability
- **D-11:** Python `logging` → stderr (lands in the systemd journal for free under
  Phase 4's root unit). INFO progress lines every N dirs / N seconds; **ETA from
  dirs-scanned rate**; one **structured summary record** at the end (dirs, duration,
  db_bytes). **stdout stays pure JSON** — observability never pollutes the JSON contract.

### Claude's Discretion
- Exact `page_size` value, progress cadence (N dirs vs N seconds), and the structured
  summary's precise field set — planner/executor may choose, consistent with D-10/D-11.

</decisions>

  — keep; just swap BLOB values for `path_id`/`parent_id` ints.

  `parent_path` as stored data is redundant beyond the diff `COALESCE`.

- `SCHEMA_VERSION` + `initialize_database` rebuild path — bump version; **no migration
  shim**, fresh schema (no backward compat; dev DBs regenerate).
- Reporting layer materializes path bytes and does prefix/ancestor matching in Python,
  not SQL — this is WHY flat dictionary wins over tree.

### Integration Points
- Scanner → `insert_directory_rows`: the dedup cache + `paths` upsert sits here.
- Every reporting/diagnostics SELECT that currently reads `path`/`parent_path` gains a
  `JOIN paths`.

  number — the headline reduction is a function of path stability and must be falsifiable.
- The 3–5 spaced real scans (C) produce the **churn measurement**, which is both a
  benchmark input and the ROI-determining unknown (see open item below).

---

## ⚠️ ROI-determining open item (panel-flagged, carry into planning)

All three deep experts independently landed on the same unknown: **the real path
cardinality / churn across the retention window** (distinct paths vs total rows). That
ratio is the entire ROI — high stability ⇒ ~5–6× win, phase clearly worth it; high churn
⇒ win collapses toward just index int-ization. **This is the first deliverable of the
benchmark** (the 3–5 spaced real scans), and it feeds the D-09 gate. Planning should
sequence the churn measurement early so the size target is set against real data.
